# 4.6. Estructura de Carpetas del Proyecto

## Objetivo

Definir y documentar la estructura de carpetas del proyecto **M3Motors** para Backend y Frontend, siguiendo los principios de **Arquitectura Limpia (Clean Architecture)** y **Domain-Driven Design (DDD)**. Esta estructura garantiza la separación de responsabilidades, mantenibilidad y escalabilidad del código a lo largo del ciclo de vida del proyecto.

---

## Backend — Clean Architecture + DDD

El backend se organiza en cuatro capas principales siguiendo el modelo de **Arquitectura Limpia**, complementado con principios de **DDD** para el modelado del dominio. Las dependencias fluyen hacia adentro: **Interfaces → Infraestructura → Aplicación → Dominio**. El dominio es la capa más interna y no tiene dependencias de otras capas.

```
backend/
├── src/
│   ├── domain/                     # Capa de Dominio (núcleo del negocio)
│   │   ├── entities/               # Entidades del dominio con identidad propia
│   │   ├── value-objects/          # Objetos de valor inmutables
│   │   ├── aggregates/             # Agregados que agrupan entidades y VO
│   │   ├── domain-services/        # Lógica de dominio que no pertenece a una entidad
│   │   ├── ports/                  # Interfaces (contratos) para capas externas
│   │   │   ├── repositories/       # Contratos de repositorios
│   │   │   ├── events/            # Contratos de eventos de dominio
│   │   │   └── services/          # Contratos de servicios externos
│   │   └── events/                 # Eventos de dominio
│   │
│   ├── application/                # Capa de Aplicación (casos de uso)
│   │   ├── use-cases/              # Casos de uso orquestan el flujo de negocio
│   │   ├── dto/                    # Data Transfer Objects
│   │   ├── handlers/               # Handlers para comandos y consultas (CQRS)
│   │   └── event-listeners/        # Suscriptores de eventos de dominio
│   │
│   ├── infrastructure/             # Capa de Infraestructura (adaptadores)
│   │   ├── persistence/            # Implementaciones de persistencia
│   │   │   ├── repositories/       # Implementaciones concretas de repositorios
│   │   │   └── migrations/         # Migraciones de base de datos
│   │   ├── external-services/      # Integraciones con servicios externos (API, SDK)
│   │   ├── messaging/              # Colas, eventos, brokers (RabbitMQ, Kafka)
│   │   ├── auth/                   # Implementación de autenticación/autorización
│   │   ├── logging/                # Configuración y adaptadores de logging
│   │   └── config/                 # Configuración global (variables de entorno)
│   │
│   ├── interfaces/                 # Capa de Interfaces (entry points)
│   │   ├── http/                   # Interfaces HTTP
│   │   │   ├── controllers/        # Controladores que reciben requests
│   │   │   ├── middlewares/        # Middlewares (auth, validación, logging)
│   │   │   ├── routes/            # Definición de rutas (Router)
│   │   │   └── validators/        # Validación de esquemas de entrada
│   │   ├── webhooks/               # Manejadores de webhooks
│   │   └── events/                 # Manejadores de eventos entrantes
│   │
│   └── shared/                     # Código compartido transversal
│       ├── errors/                 # Clases de error personalizadas
│       ├── utils/                  # Utilidades y helpers genéricos
│       └── constants/              # Constantes globales
│
├── tests/                          # Pruebas automatizadas
│   ├── unit/                       # Pruebas unitarias
│   ├── integration/                # Pruebas de integración
│   └── e2e/                        # Pruebas end-to-end
│
├── .env.example                    # Plantilla de variables de entorno
├── package.json                    # Dependencias y scripts del proyecto
├── tsconfig.json                   # Configuración de TypeScript
└── README.md                       # Documentación del backend
```

### Propósito de cada carpeta — Backend

| Carpeta | Propósito |
|---|---|
| `domain/entities/` | Entidades con identidad propia (ej: `Vehiculo`, `Cliente`, `OrdenTrabajo`). |
| `domain/value-objects/` | Objetos inmutables sin identidad (ej: `Email`, `RUC`, `Placa`, `Dinero`). |
| `domain/aggregates/` | Agrupaciones transaccionales de entidades y VO (ej: `Presupuesto`). |
| `domain/domain-services/` | Lógica de negocio que no pertenece a una sola entidad. |
| `domain/ports/` | Interfaces que definen contratos para la infraestructura. |
| `application/use-cases/` | Orquestación de flujos de negocio (ej: `CrearOrdenTrabajoUseCase`). |
| `application/dto/` | Objetos para transferencia de datos entre capas. |
| `infrastructure/persistence/` | Implementación de acceso a datos (TypeORM, Prisma, MongoDB). |
| `infrastructure/external-services/` | Integración con APIs de terceros (SRI, proveedores, notificaciones). |
| `interfaces/http/controllers/` | Puntos de entrada HTTP que reciben requests y delegan en casos de uso. |
| `interfaces/http/middlewares/` | Filtros HTTP (autenticación, logs, rate-limiting, CORS). |

### Principios aplicados

- **Inversión de dependencias**: Las capas externas dependen de abstracciones definidas en el dominio.
- **Aislamiento del dominio**: El código de negocio no depende de frameworks, bases de datos ni librerías externas.
- **Separación de responsabilidades**: Cada carpeta tiene un rol claramente definido y no mezcla responsabilidades.
- **Escalabilidad por módulos**: Para proyectos grandes, cada capa puede subdividirse por módulos de dominio (ej: `domain/vehicle/entities/`, `domain/maintenance/entities/`).

## Frontend — Atomic Design + Componentes

El frontend se organiza siguiendo el patrón **Atomic Design** para componentes y una estructura modular para servicios y estado. Esto permite escalar la interfaz de usuario de manera consistente y reutilizable.

```
frontend/
├── src/
│   ├── components/                 # Componentes reutilizables (Atomic Design)
│   │   ├── atoms/                  # Componentes básicos e indivisibles
│   │   ├── molecules/              # Combinaciones de átomos
│   │   ├── organisms/              # Componentes complejos con lógica propia
│   │   └── templates/              # Layouts y estructuras de página
│   │
│   ├── pages/                      # Pantallas y vistas del Dashboard
│   ├── hooks/                      # Custom hooks de React
│   ├── services/                   # Consumo de APIs del backend
│   ├── store/                      # Estado global (Context API, Zustand, Redux)
│   ├── utils/                      # Utilidades y funciones helper
│   │
│   ├── types/                      # Definiciones de tipos TypeScript
│   ├── styles/                     # Estilos globales, temas, animaciones
│   │
│   └── assets/                     # Recursos estáticos
│       ├── images/                 # Imágenes y gráficos
│       ├── icons/                  # Íconos SVG/componentes
│       └── fonts/                  # Tipografías
│
├── public/                         # Archivos públicos estáticos
├── tests/                          # Pruebas del frontend
│
├── .env.example                    # Variables de entorno de ejemplo
├── package.json                    # Dependencias y scripts
├── tsconfig.json                   # Configuración de TypeScript
└── README.md                       # Documentación del frontend
```

### Propósito de cada carpeta — Frontend

| Carpeta | Propósito |
|---|---|
| `components/atoms/` | Componentes básicos reutilizables (Botón, Input, Etiqueta, Selector, Icono). |
| `components/molecules/` | Grupos de átomos con funcionalidad específica (CampoFormulario, BarraBúsqueda, Tarjeta). |
| `components/organisms/` | Componentes complejos como Tabla, Formulario, Header, Sidebar, Modal. |
| `components/templates/` | Layouts que definen la estructura de las páginas (DashboardLayout, AuthLayout). |
| `pages/` | Páginas completas que combinan templates y organismos (Dashboard, Login, Vehículos). |
| `hooks/` | Lógica reutilizable encapsulada en hooks (useAuth, useVehiculos, usePaginación). |
| `services/` | Módulos de comunicación con la API del backend (apiClient, authService, vehiculoService). |
| `store/` | Gestión de estado global de la aplicación. |
| `utils/` | Funciones utilitarias (formateo de fechas, validaciones, constantes). |
| `types/` | Interfaces y tipos compartidos de TypeScript. |
| `styles/` | Estilos globales, temas, animaciones y variables CSS. |
| `assets/` | Recursos estáticos como imágenes, íconos y fuentes. |

### Atomic Design aplicado

El **Atomic Design** nos permite construir una interfaz consistente y escalable:

1. **Átomos**: Componentes más pequeños e indivisibles (`Button`, `Input`, `Label`, `Icon`).
2. **Moléculas**: Combinación de átomos con un propósito (`SearchBar = Input + Button`, `FormField = Label + Input + Error`).
3. **Organismos**: Componentes complejos que pueden contener moléculas y átomos (`DataTable`, `VehicleForm`, `Sidebar`).
4. **Plantillas**: Layouts que organizan organismos en una estructura de página (`DashboardLayout`, `AuthLayout`).

## Convenciones generales

### Archivos .gitkeep

Se incluye un archivo `.gitkeep` en cada directorio vacío para que Git mantenga la estructura de carpetas en el repositorio. Estos archivos deben ser eliminados cuando el directorio contenga al menos un archivo de código.

### Nomenclatura de archivos

| Tipo | Convención | Ejemplo |
|---|---|---|
| Clases/Entidades | PascalCase | `Vehiculo.ts`, `OrdenTrabajo.ts` |
| Casos de uso | PascalCase + UseCase | `CrearVehiculoUseCase.ts` |
| Controladores | PascalCase + Controller | `VehiculoController.ts` |
| Componentes React | PascalCase | `DataTable.tsx`, `SearchBar.tsx` |
| Hooks | camelCase + use | `useVehiculos.ts`, `useAuth.ts` |
| Servicios | camelCase + Service | `vehiculoService.ts` |
| Utilidades | camelCase | `formatDate.ts`, `validators.ts` |
| DTOs | PascalCase + DTO | `CreateVehiculoDTO.ts` |

### Organización por módulos (escalabilidad)

Para proyectos con múltiples contextos delimitados (Bounded Contexts), se recomienda extender la estructura organizando por módulos dentro de cada capa:

```
backend/src/domain/vehiculo/entities/
backend/src/domain/vehiculo/value-objects/
backend/src/domain/mantenimiento/entities/
backend/src/application/vehiculo/use-cases/
backend/src/application/mantenimiento/use-cases/
```

Esta organización se adopta cuando la complejidad del dominio lo requiera, manteniendo la misma arquitectura base.

## Criterios de Aceptación

| ID | Criterio |
|---|---|
| AC-01 | La estructura de carpetas del Backend sigue los principios de Clean Architecture (dominio → aplicación → infraestructura → interfaces). |
| AC-02 | La estructura de carpetas del Frontend está alineada con Atomic Design y la guía de componentes (átomos, moléculas, organismos). |
| AC-03 | El documento describe claramente el propósito de cada carpeta y las convenciones de nomenclatura. |
| AC-04 | La estructura contempla escalabilidad mediante organización por módulos y separación de responsabilidades. |
